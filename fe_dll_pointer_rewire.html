科目B ─ 双方向リスト / 矢印の付け替え 激ムズ

REWIRE

クラス定義と前提

双方向リストの各要素は次のクラス Node で表す。(値・前後2つの参照を持つ)

クラス Node 値 : 整数 前 : Node // 1つ前の要素への参照 次 : Node // 1つ次の要素への参照

このリストは両端に番兵(ばんぺい)を持つ。先頭番兵 先頭 と末尾番兵 末尾 は値を持たないダミー要素で、実データはその間に並ぶ。すべての実ノードは必ず前と次を持つ(端でも 未定義 にならない)ため、ヌル判定が不要になるのが番兵方式の利点。

上段の矢印が「次」、下段が「前」。実ノードは N1〜N3。

設問

手続 move(curr, target) は、リスト内の実ノード curr を取り外し、別の実ノード target の直後へ移動する。(curr と target は異なる実ノードとする)

空欄 a〜d を埋め、さらに最後の2行の正しい順序を選べ。

○手続: move(Node: curr, Node: target) // 手順1: curr をリストから切り離す curr.前.次 ← // a curr.次.前 ← // b // 手順2: curr を target の直後へ挿入する curr.次 ← // c curr.前 ← // d // ↓ この2行の順序が問題（設問の選択で確定） （順序を下で選択）

最後の2行の正しい順序は？

① target.次.前 ← curr　→　target.次 ← curr
② target.次 ← curr　→　target.次.前 ← curr

解答・解説

a = curr.次　　b = curr.前　　c = target.次　　d = target
順序 = ①（target.次.前 ← curr を先、target.次 ← curr を後）

① 手順1：切り離し（番兵があるのでヌル判定ゼロ）

curr の左隣と右隣を直結する。curr.前.次 ← curr.次 で左隣の「次」を curr の右隣へ。curr.次.前 ← curr.前 で右隣の「前」を curr の左隣へ。番兵のおかげで curr が端でも左隣・右隣は必ず存在するので、「もし 未定義 なら」の場合分けが要らない。これが番兵方式の最大の旨味。

② 手順2：挿入（target と 旧target.次 の間に割り込む）

つなぎ替える矢印は4本。curr.次 ← target.次(旧)、curr.前 ← target、target.次.前 ← curr、target.次 ← curr。

target と 旧target.次 の間に curr を割り込ませる4本

③ なぜ順序①でないと壊れるか（ここが本問の急所）

罠： target.次 ← curr を先にやると、その瞬間 target.次 は curr 自身を指す。続く target.次.前 ← curr は実質 curr.前 ← curr になり、curr が自分自身を前に持つ壊れ方をする。

さらに嫌らしいのは、この壊れ方は「次」をたどる前向き走査では気づけないこと。前向きには正しく並んで見えるのに、「前」をたどる後ろ向き走査だけが破綻する。実際に move(N3, target=N1) を順序②でやると、前向きは N1,N3,N2,… と正しく見えるのに、後ろ向きでは N3 が消える（N3.前 が N3 自身を指すため)。双方向リストのバグが見つけにくい典型。

だから target.次(旧値)を参照し終えるまで target.次 を書き換えてはいけない。順序①では、target.次.前 ← curr の時点ではまだ target.次 が旧値なので正しく旧ノードの「前」を直せる。最後に target.次 ← curr で締める。

実務的な安全策： 旧値を一時変数に退避すれば順序に悩まなくて済む。
tmp ← target.次　/　curr.次 ← tmp　/　curr.前 ← target　/　tmp.前 ← curr　/　target.次 ← curr
「参照を上書きする前に、必要な旧値を退避する」── これがポインタ付け替え全般の鉄則。試験でも、どの参照がいつ書き換わるかを1行ずつ追えば順序トラップは必ず見抜ける。

④ 手順1と手順2の順序

切り離し(手順1)は curr の旧前後を使う。挿入(手順2)で curr.前・curr.次 を上書きしてしまうので、必ず切り離しを先に。逆にすると旧リンクを失って切り離せなくなる。
